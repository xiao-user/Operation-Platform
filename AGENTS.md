# 项目协作规范

本文件适用于仓库根目录及全部子目录。项目是 Vue 3 + TypeScript 的多租户 SaaS 管理前端，当前使用 `localStorage` 完成组织、菜单、工作台和角色权限的原型闭环；前端权限不能视为生产安全边界。

## 开始修改前

- 先阅读 `README.md`，确认当前产品边界、数据模型和验证命令。
- 保留工作区中与任务无关的已有修改，不重置、不覆盖用户文件。
- 设计文档和实施计划记录早期决策；当它们与 README、测试或当前代码冲突时，以当前代码和测试为准，并同步修正文档。
- 不在视图组件中新增租户、菜单、角色或页面的硬编码副本，应使用现有配置、Store 和 repository。

## 核心架构约束

- `src/config/page-registry.ts` 是内部页面资源、路径和动态路由的代码级事实源。新增真实页面时先实现组件，再注册唯一 `pageKey`；菜单不得保存 Vue 组件路径。
- `src/config/menu-template-definitions.ts` 和 `src/config/menu-templates.ts` 只用于首次初始化与恢复默认，不参与运行时导航。模板变更不得静默覆盖已配置租户。
- 普通菜单最多四级：一级模块 → 二级目录 → 三级目录 → 内部页面或外部链接。层级规则统一由 `MAX_MENU_DEPTH` 和领域校验维护，不在组件中复制判断。
- 工作台属于租户 Shell 配置，不是普通菜单节点；它不能拖入菜单树或绑定业务页面。
- 菜单、工作台和角色必须通过 `src/features/tenant-config/` 聚合仓库原子读写。禁止在新增流程中分别持久化三个旧 key；旧仓库仅保留迁移兼容职责。
- 组织数据通过 `src/features/tenant/` 仓库维护。删除租户时必须同步清理租户聚合配置，并保留失败回滚语义。
- repository 必须返回防御性副本并显式抛出持久化错误。损坏数据只有在原始值备份成功后才能恢复默认，禁止静默吞错或覆盖。

## 页面、菜单与权限

- 删除菜单只删除导航绑定，不删除页面资源或页面代码；重新创建菜单时可再次关联页面注册表中的资源。
- 尚未开发的菜单使用 `developing-placeholder` 统一缺省页。该资源允许重复绑定，并通过菜单 ID 形成独立地址。
- RBAC 默认拒绝：没有有效租户角色时不展示菜单，也不允许路由访问。
- 管理员角色固定拥有当前租户全部可见叶子菜单，不在菜单配置中重复勾选；`requiresAdmin` 仍是系统管理页的硬限制。
- 普通角色只授权内部页面或外部链接叶子节点；父级模块与目录根据获权子节点自动保留。
- 菜单显隐是租户级总开关，角色授权是二次过滤。路由守卫与导航必须使用同一套过滤语义。
- 当前角色来自认证会话的租户角色映射，不在 Header 中添加客户端角色切换来模拟授权。

## 导航与布局

- `src/layouts/AppLayout.vue` 是持续挂载的应用 Shell。工作台无侧栏；业务模式由一级模块栏、顶部二级导航、深层侧栏和 `RouterView` 组成。
- Vue Router 是当前页面和导航高亮的事实源。导航前不得预先清空当前模块或菜单状态，以免出现闪烁和短暂空白。
- 内部导航使用 `RouterLink` 或 Router API；只有外部链接使用 `<a>`/`window.open`，并遵循配置的打开方式。
- 顶部 Header 保持挂载并更新内容。当前只为一级模块栏的进入与退出保留短促过渡；侧栏、顶部导航文字和业务页面不增加路由切换动画，除非需求明确改变。
- 可见样式使用 `src/styles` 中的设计变量，避免在业务组件中新增可复用颜色、间距和尺寸的魔法值。

## 菜单配置交互

- 菜单名称、上级/关联目标、图标和排序支持点击或双击进入行内编辑；`Enter` 保存，`Escape` 取消，抽屉编辑入口继续保留。
- 拖拽只在完整树上启用；筛选状态下暂停拖拽。前后插入使用细线指示，跨级放入使用目标行状态，不添加说明文案或持久选中背景。
- 拖拽、编辑和保存后不得自动展开节点；展开状态由用户手动控制。
- 页面下拉选项来自页面注册表，并显示资源状态。不要在菜单表单中恢复可编辑路由或组件路径。
- 图标选择使用现有 Lucide 动态加载和虚拟化选择器，不要整体导入图标库或为少量菜单复制一套图标枚举。

## 测试与完成标准

常用命令：

```bash
npm install
npm run dev
npm run check
npm audit --offline
```

- 提交前运行 `npm run check`，它依次执行 Oxlint、ESLint、应用与测试类型检查、Vitest 和生产构建。
- 修改菜单树、仓库、权限或路由时，必须补充对应单元测试；修复回归时先添加能复现问题的测试。
- 修改可见交互时，除自动化检查外还应在浏览器验证目标流程。当前仓库未配置自动化 E2E，不要把单元测试描述为端到端验证。
- 生产构建不应出现已知的大 chunk 警告；新增依赖或静态全量导入前先评估拆包影响。
- GitHub Actions 在 push 和 pull request 时运行 `npm run check` 以及生产依赖审计。

## Git 与文档

- 提交应按单一职责拆分，提交信息说明实际变更，不混入无关格式化。
- 不使用破坏性 Git 命令处理已有修改；合并到主分支前保证工作区干净并运行完整检查。
- 架构边界、命令、入口、存储 key 或已实现能力变化时，同步更新 `README.md`；影响协作约束时同步更新本文件。
